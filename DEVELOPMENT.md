# zmenu Development SOP

Standard operating procedure for modifying `zmenu.sh`.  
**Current version:** 5.11.0  
**Source file:** `~/projects/zmenu/zmenu.sh`  
**Live binary:** `/usr/local/bin/zmenu`

---

## The Golden Rule

Always edit the **source file** (`~/projects/zmenu/zmenu.sh`).  
Never edit `/usr/local/bin/zmenu` directly — it gets overwritten on every install.

---

## Architecture — Three Layers

Before editing anything, understand which layer you're working in:

```
Discovery (live shell commands)  →  Wiki (persistent knowledge)  →  AI (interprets)
```

**Discovery layer** (`_disc_*` functions, `D_*` vars): runs live shell commands to find what IS there right now. No opinions, no hardcoded tool names, no "this is deprecated" logic. If it can be discovered with a shell command, it goes here.

**Wiki layer** (`~/.zmenu/wiki/*.md`): captures structured facts about this specific machine. Generated by `_wiki_full_refresh` at startup and on-demand. Grows over time. This is Zenny's knowledge of the system — not hardcoded shell conditionals.

**AI layer** (`_ctx_*` + `_cc_inline`): interprets wiki context. Knows what's deprecated, what's risky, what to recommend. This is the AI's job — never write shell conditionals to simulate AI judgment.

**Rule:** if a shell command can discover it → wiki. If it requires knowing what something *means* → AI. Never hardcode package-specific knowledge in display functions.

## Variable Discipline

### Config variables (`cfg_*` / `ZMENU_*` prefix)
Defined at top of file, user-overridable via `~/.zmenu/config`:

| Variable | Default | Purpose |
|----------|---------|---------|
| `ZMENU_VERSION` | current | readonly, bump on each release |
| `ZMENU_SELF` | `$(realpath ...)` | readonly, script's own path |
| `ZMENU_INSTALL_PATH` | `/usr/local/bin/zmenu` | install destination |
| `ZMENU_CONFIG_DIR` | `~/.zmenu` | user config root |
| `ZMENU_WIKI_DIR` | `~/.zmenu/wiki` | wiki markdown files |
| `ZMENU_PROJECTS_DIR` | `~/projects` | scanned by Projects module |
| `ZMENU_AI_BACKEND` | `auto` | `auto\|zenny\|opencode\|ollama` |
| `ZMENU_AI_MODEL` | `""` | empty = auto-select |
| `ZMENU_AI_CONTEXT_LENGTH` | `8192` | tokens sent with each request |
| `ZMENU_ZENNY_BINARY` | `~/.local/bin/zenny-core` | path to local inference binary |
| `ZMENU_GPU_GFX_OVERRIDE` | `""` | force a specific gfx ID |
| `ZMENU_MACHINE_LABEL` | `""` | shown in AI prompts (defaults to hostname) |
| `ZMENU_PREFERRED_EDITOR` | `$VISUAL\|$EDITOR\|nano` | editor for in-menu editing |

### Process/path variables (runtime, not in config)
These must be referenced by variable — never hardcoded as string literals:

| Variable | Value | Used for |
|----------|-------|---------|
| `ZENNY_BINARY` | from `ZMENU_ZENNY_BINARY` | path to binary |
| `ZENNY_PROCESS` | `zenny-core` | `pgrep "$ZENNY_PROCESS"` |
| `ZENNY_LOG` | `/tmp/zenny-core.log` | log file |
| `ZENNY_SOCKET` | `/tmp/zenny-core.sock` | socket path |
| `OPENCODE_BIN` | `~/.opencode/bin/opencode` | fallback path |
| `OPENCODE_PROCESS` | `opencode` | `pgrep "$OPENCODE_PROCESS"` |

**Rule:** never write `pgrep zenny-core` or `pkill opencode` as literals.  
Write `pgrep "$ZENNY_PROCESS"` and `pkill "$OPENCODE_PROCESS"` instead.

### Discovery variables (`D_*` prefix)
All populated by `_disc_*` functions during `discover()`. Do not add new ones without a corresponding `_disc_*` function.

## Module Pattern — Every Section Must Have Five Components

Every top-level module follows this exact pattern. Recognising it prevents breaking things:

| Component | Naming | Purpose |
|-----------|--------|---------|
| Menu loop | `mod_<name>()` | `while` loop, `header`, options display, `case` handler |
| Context fn | `_ctx_<name>()` | Generates live text for AI system prompt |
| Apply fn | `_apply_<name>()` | Calls `_apply_generic "$1" "<Name>"` |
| Wiki file | `<name>.md` in `ZMENU_WIKI_DIR` | Generated by `_wiki_full_refresh` |
| C) binding | `C) _cc_inline "<Name>" _ctx_<name> _apply_<name>; pause ;;` | In menu case handler |

### Menu loop skeleton

```bash
mod_example() {
    while true; do
        header                                    # clears screen + prints banner
        echo -e "${BCYN}┄ TITLE${NC}"            # section header (cyan, always this style)
        echo ""
        # ... display state ...
        echo "   1)  Option one"                  # menu options (3 spaces indent)
        echo "   2)  Option two"
        echo ""
        echo "   C)  ✦ Ask AI    r)  Back"
        echo ""
        read -rp "  Selection: " ch              # input prompt
        case $ch in
            1) _example_do_thing; pause ;;        # calls helper, then pauses
            2) _example_other; pause ;;
            C) _cc_inline "Example" _ctx_example _apply_example; pause ;;
            r|R) break ;;                         # always have this
            *) echo -e "${RED}  Invalid.${NC}"; sleep 1 ;;
        esac
    done
}
```

### Helper functions (no while loop)

```bash
_example_do_thing() {
    header
    echo -e "${BCYN}┄ DO THING${NC}"
    echo ""
    # ... do the work ...
    echo -e "  ${OK}  Done"
}
```

## Colour and Symbol Reference

```bash
${BCYN}text${NC}    # Cyan  — section headers, highlighted values
${BOLD}text${NC}    # Bold  — field labels
${DIM}text${NC}     # Dim   — hints, secondary info, notes
${BRED}text${NC}    # Red   — destructive actions (Stop, Delete)

$OK                 # Green  ●  running / healthy / success
$FAIL               # Red    ●  error / stopped / critical
$WARN               # Yellow ●  warning / attention
$IDLE               # Grey   ○  installed but not running / off
```

### Status display rule — installed ≠ running

For any tool that runs as a daemon, `$OK` means **process is running** (pgrep confirms), not just installed.

| State | Indicator |
|-------|-----------|
| `pgrep -x "$PROCESS_VAR"` returns 0 | `$OK` green dot |
| Installed but not running | `$IDLE` grey + "(not running)" |
| Not installed | `$IDLE` grey + "not installed" |
| Error / misconfigured | `$WARN` or `$FAIL` |

Applies to: Zenny-Core, Ollama, OpenCode, LM Studio, and any future daemon.

## Apply Engine Safety Rules

`_apply_generic` uses an **allowlist** approach — nothing runs unless explicitly permitted.

### Allowlist (commands that may be extracted and run)
```
systemctl, sysctl, docker, pkill, kill, killall (not targeting zmenu/bash),
apt, snap, mkdir, rm (no -rf /), cp, mv, chmod, chown,
tee, export, python3, pip, curl, cat, echo, powerprofilesctl,
cpupower, journalctl, sed, awk
```

### Hard blocks (rejected regardless of context)
```
pkill/killall/kill targeting: zmenu, bash, current $BASHPID
Any command launching zenny-core directly in background (duplicate instance risk)
Re-launching zmenu itself from within apply
sudo rm -rf  (destructive)
Commands containing $(...) or backtick substitution (injection risk)
```

### Implementation pattern
1. Extract commands from ` ```bash ` fenced blocks first (preferred), then bare-prefix fallback
2. Validate each extracted command against allowlist before eval
3. Block + log any matching hard-block patterns
4. After kill/pkill: verify with pgrep before reporting success

## AI Routing Rules

| Scenario | Use | Why |
|----------|-----|-----|
| `C) Ask AI` in any section | `_cc_inline` | Inline chat, respects `AI_BACKEND_ACTIVE` |
| User wants interactive coding session | `cc_launch` + OpenCode TUI | Full TUI experience |
| Backend = OpenCode when `_cc_inline` called | Show error, direct to Settings | OpenCode is TUI-only, not inline |
| Starting/stopping AI engines | `_zenny_start/_stop`, `_opencode_stop` | Always verify with pgrep after |

**Never** add a `C) cc_launch` binding — all Ask AI bindings use `_cc_inline`.

### Zenny-Core specific
- Start: use `_zenny_start()` — does live pgrep check before launching
- Stop: use `_zenny_stop()` — kills by PID if known, pkill by name as fallback
- Never start zenny-core directly in `_apply_generic` — creates duplicate instances
- Double-instance check: `pgrep -x "$ZENNY_PROCESS"` before any start attempt

## System Scan Registry

`mod_system_scan` is driven by `_SCAN_REGISTRY`. Adding a new app = one line.

**Format:**
```
"DisplayName|installed_check|process_name|service_name|port|config_path|category|description"
```

| Field | Values |
|-------|--------|
| `installed_check` | absolute path, `~` path, command name, or `docker:container_name` |
| `process_name` | used with `pgrep -x` |
| `port` | number, or `unix:socket_path` |
| `category` | `ai-inference` \| `ai-tools` \| `networking` \| `containers` \| `dev` |

The scanner, drill-down, wiki generation, and AI context all pick up new entries automatically.

## Version Bump Policy

| Change scope | Bump |
|---|---|
| Typo, cosmetic, wording | patch (`5.11.x`) |
| New option in existing menu, new helper function | minor (`5.x.0`) |
| New top-level module, new config var, architectural pass | minor (`5.x.0`) |
| Breaking change to config format | major (`x.0.0`) |

One version bump per logical change set — not per individual edit. Bump the line:
```bash
readonly ZMENU_VERSION="X.Y.Z"   # near the top of zmenu.sh
```

## Phase 1 — Before You Touch Anything

### Step 1.1 — Backup

Run this every time, no exceptions:

```bash
ZMENU_SRC=~/projects/zmenu/zmenu.sh
BACKUP_DIR="$(dirname "$ZMENU_SRC")/.zmenu-backups"
mkdir -p "$BACKUP_DIR"
cp "$ZMENU_SRC" "$BACKUP_DIR/zmenu-$(date +%Y%m%d-%H%M%S).sh"
cp /usr/local/bin/zmenu "$BACKUP_DIR/zmenu-installed-$(date +%Y%m%d-%H%M%S).sh"
# Keep only the 20 most recent backups
ls -t "$BACKUP_DIR"/zmenu-*.sh | tail -n +21 | xargs rm -f 2>/dev/null || true
echo "Backup done → $BACKUP_DIR"
```

Verify:
```bash
ls -lh ~/projects/zmenu/.zmenu-backups/ 2>/dev/null || ls -lh ~/projects/zmenu/.zmenu-backups/
```

### Step 1.2 — Check version alignment

```bash
grep 'ZMENU_VERSION' ~/projects/zmenu/zmenu.sh | head -1
grep 'ZMENU_VERSION' /usr/local/bin/zmenu | head -1
```

Both should match. If source is ahead, that's fine — it just hasn't been installed yet.

### Step 1.3 — Read the section you're changing

Never add code without first reading what's already there. Search before writing:

```bash
# Find by function name:
grep -n 'function_name' ~/projects/zmenu/zmenu.sh

# Find by screen text:
grep -n 'text you see' ~/projects/zmenu/zmenu.sh

# Find existing patterns similar to what you want to add:
grep -n '_disc_\|D_GPU\|_ctx_hardware' ~/projects/zmenu/zmenu.sh | head -20
```

## Phase 2 — Making the Edit

### Step 2.1 — Open the source file

```bash
# nano (simplest):
nano ~/projects/zmenu/zmenu.sh

# preferred editor:
code ~/projects/zmenu/zmenu.sh
nvim ~/projects/zmenu/zmenu.sh
```

**In nano:**
- `Ctrl+W` → search for function name or visible text → Enter
- `Ctrl+W` again → next occurrence
- `Ctrl+_` → jump directly to a line number

### Step 2.2 — Edit discipline

- Apply changes to **all affected sections** — if a pattern changes, change it everywhere
- Search for existing similar functions before writing new ones
- For new discovery variables: add the `D_*` declaration near line 150, add the `_disc_*` function in §2, call it from `discover()`
- For new wiki sections: add to `_wiki_full_refresh()` AND `_wiki_path()`

### Step 2.3 — Common pitfalls

**"unbound variable" error** (`set -u` is active):
```bash
# WRONG — crashes if VAR is not set:
echo "$SOME_VAR"

# RIGHT — safe with fallback:
echo "${SOME_VAR:-}"          # empty fallback
echo "${SOME_VAR:-default}"   # explicit fallback
```

**Function silently does nothing / exits early** (`set -e` is active — any non-zero exit code kills the function):
```bash
# WRONG — if somecommand fails, whole script dies:
somecommand

# RIGHT — suppress failure:
somecommand 2>/dev/null || true
somecommand 2>/dev/null || echo "failed, continuing"
result=$(somecommand 2>/dev/null || echo "")
```

**Menu option does nothing when pressed** — check three things:
1. The `case` letter matches exactly — `r` and `R` are different characters; use `r|R)`
2. The function name is spelled correctly — typos are silent in bash
3. The function actually exists: `grep -n '^_function_name' ~/projects/zmenu/zmenu.sh`

**Discovery data is stale:**
```bash
# From inside zmenu: Settings → e) Re-run discovery

# Debug a specific discovery function:
bash --norc -c 'source ~/projects/zmenu/zmenu.sh; discover; echo "GPU: $D_GPU_GFX"'
```

## Phase 3 — Save and Install

### Step 3.1 — Syntax check (always, before install)

```bash
bash -n ~/projects/zmenu/zmenu.sh && echo "✓ Syntax OK" || echo "✗ SYNTAX ERROR — do not install"
```

```bash

echo "✓ Installed"

```bash

## Phase 4 — When Things Go Wrong

### zmenu won't start / crashes immediately

```bash
# Run with bash trace mode to see exactly what fails:
bash -x /usr/local/bin/zmenu 2>&1 | head -50
```

Look for the last line before the error — that's where it died.

### Something broke, need to roll back

```bash
# List backups (newest first):
ls -lt ~/projects/zmenu/.zmenu-backups/

# Roll back source to a specific backup:
cp ~/projects/zmenu/.zmenu-backups/zmenu-20260325-143022.sh ~/projects/zmenu/zmenu.sh

# Roll back the live binary directly (fastest recovery):
sudo cp ~/projects/zmenu/.zmenu-backups/zmenu-installed-20260325-143022.sh /usr/local/bin/zmenu
sudo chmod +x /usr/local/bin/zmenu

# Confirm rollback:
grep 'ZMENU_VERSION' /usr/local/bin/zmenu
zmenu
```

### Asking Claude Code to help — what to give it

1. The **exact function name(s)** you want changed: `grep -n 'function_name' zmenu.sh`
2. The **line range** of the relevant block (e.g. "lines 1342–1577")
3. **What you want it to do** — be specific
4. The **current version** so it knows what bump to apply

Claude will read the live file, produce the full replacement block, and tell you exactly which lines to replace. You never need to paste the whole file.
